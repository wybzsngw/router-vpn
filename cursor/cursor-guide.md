## ① Cursor 是什么

Cursor 可以理解为“把 AI 深度融合进工程工作流”的编程环境。它不是单纯补全插件，而是把多个交互入口串成闭环：

1. Tab：光标处预测续写，处理高频重复动作；
2. Cmd+K：选中即改，处理小范围重构与定向修复；
3. Chat：做方案推演、错误定位、任务拆解；
4. Composer/Agent：处理跨文件任务、批量修改和验证回路。

Cursor 官方在模型体系里明确区分了两类用量池：一类是 Auto/Composer 池，另一类是按模型 API 计费的池。你可以把它理解为“高频日常任务”和“指定强模型任务”两条成本轨道，这也是后文我们建议“分阶段选模型”的原因。

对中国用户最实用的理解是：Cursor 的价值不只是“生成代码”，而是把需求澄清、实现、验证、回归这四个动作放在同一个上下文里，减少反复切窗和上下文丢失。

### 这篇 Pillar 适合谁

- **刚接触 Cursor 的新手**：优先读 ①~⑥，先把注册、网络、价格和模型搞清楚；
- **已经在用但体验不稳**：直接跳 ⑯~⑳，把故障处理流程先建立起来；
- **想把 Cursor 用在真实交付**：重点读 ⑦~⑮，再结合文末实战延伸。

### 一句话记忆版

你可以把 Cursor 理解为“一个会持续参与你编码过程的协作副驾”：

- Tab 负责高频细粒度动作；
- Cmd+K 负责局部改造；
- Chat 负责推理和策略；
- Composer/Agent 负责批量执行和闭环验证。

## ② 下载与注册

### 先做最短闭环（10~20 分钟）

1. 下载并安装对应系统版本；
2. 完成邮箱注册/登录；
3. 首次进入设置页，确认模型列表、套餐信息、账号状态；
4. 打开一个小项目，分别试一次 Tab、Cmd+K、Chat。

### 首次上手建议

- Windows：优先保证系统与网络稳定，避免安装器中断；
- macOS：首次安全权限弹窗按系统指引放行；
- Linux：先确认 `git`、`ssh`、`node` 等常用工具在 PATH 中可用。

### 把体验“拉稳”的 5 件事

1. 设置好终端默认 shell，减少命令差异；
2. 绑定 Git 账户并完成一次拉取/提交；
3. 整理常用提示词模板，固定你的指令语气；
4. 给项目建立最小规则（命名、类型、禁改目录）；
5. 先用小任务验证流程，再把大任务交给 Agent。

### 注册开通阶段的常见误区

1. **只安装不验收**：很多人装完就算完成，结果第一次正式开发才发现模型不可用。  
   建议：首次安装后立即做“三动作验收”——Tab 补全、Cmd+K 改写、Chat 提问。
2. **忽略项目级规则**：没设置最小规则时，模型输出风格会随上下文飘。  
   建议：至少写一条“禁改目录”和一条“类型约束”。
3. **一上来就跑大任务**：上下文没稳定前直接跑跨模块任务，失败率更高。  
   建议：先做小闭环，确认网络、模型、权限和验证流程都通。

## ⑥ 模型选型

Cursor 官方将 Auto 描述为在智能、成本和可靠性之间做平衡的策略入口；Composer 2 则是针对 agentic coding 强化训练的自研模型。换句话说，Auto 更像“默认调度入口”，Composer 2 是“可预期的基础能力层”。

### 选型建议：按任务分阶段

1. 需求澄清：Auto / Composer 2；
2. 关键实现：Claude 4.5 / GPT-5；
3. 回归修补：回到 Auto 或 Composer 2。

### 常见误区

- 只看“单次回答惊艳感”，忽略全日稳定性；
- 把模型上限问题误判为提示词问题；
- 长任务不拆分，导致上下文拥塞。

### 一套可复制的默认策略

- 默认用 Auto 处理常规任务；
- 关键重构/架构讨论切高智力模型；
- 每段任务都输出验证清单（测试、边界条件、回滚点）。

### 任务类型与模型组合示例

| 任务类型 | 推荐模型策略 | 说明 |
|---|---|---|
| 快速补全、样板代码 | Auto / Composer 2 | 速度和成本更平衡 |
| 中型重构（3~8 文件） | Auto 起步，必要时切 Claude/GPT | 先低成本验证路径 |
| 架构讨论、复杂排障 | Claude / GPT-5 | 推理深度通常更好 |
| 回归修补与文档整理 | Auto / Composer 2 | 稳定收尾，控制成本 |

### “模型不稳定”时先看三件事

1. 上下文是否过长、目标是否过散；  
2. 当前任务是否超出模型适用边界；  
3. 网络质量是否足以支撑当前模型。

### 模型切换决策树（可直接照用）

**第一步：任务复杂度判断**

- 小任务（单文件、短上下文）-> Auto / Composer 2  
- 中任务（3~8 文件、需要局部重构）-> Auto 起步，必要时切高阶模型  
- 大任务（跨模块、架构与回归联动）-> 先高阶模型做方案，再分批执行

**第二步：失败类型判断**

- 结果浅 -> 优先升级模型  
- 速度慢 -> 优先减小上下文与任务批次  
- 超时多 -> 优先查网络和并发

**第三步：成本判断**

- 如果你在回归阶段继续用高阶模型，成本会明显抬升；  
- 回归和文档收尾阶段切回 Auto/Composer，通常性价比更高。

### 团队模型策略模板

| 场景 | 默认模型 | 升级条件 | 回落条件 |
|---|---|---|---|
| 日常开发 | Auto | 连续两次关键任务输出不达标 | 任务回到常规改动 |
| 架构设计 | Claude / GPT-5 | 需要多轮推理与权衡 | 方案稳定后进入实现 |
| 回归修补 | Composer 2 / Auto | 发现复杂回归根因不明 | 根因明确后继续收尾 |

| Composer | 跨文件任务与批处理 | 模块级 | 分钟级到十分钟级 |

推荐工作流：先 Chat 对齐目标 -> Cmd+K 做局部验证 -> Composer 执行批改 -> Tab 收尾提速。

### 四入口分工的实战例子

假设你要“给支付模块补幂等 + 回归测试”：

1. **Chat**：先产出任务拆解与风险清单；  
2. **Cmd+K**：在单个函数里验证幂等逻辑写法；  
3. **Composer**：批量改服务层、路由层、测试层；  
4. **Tab**：收尾补注释、补重复样板、补边界分支。

这种顺序比“全程只用 Chat”更稳，因为每个入口都在做它最擅长的事情。

### 四入口映射到一天开发节奏

**上午（需求拆解期）**

- Chat：对齐目标和验收标准  
- Cmd+K：做关键函数可行性验证

**下午（主实现期）**

- Composer：批量改代码和测试  
- Tab：补样板、补注释、补重复分支

**收尾（验收期）**

- Chat：复盘风险和补测试清单  
- Cmd+K：对问题点做定向修复

把这套节奏固定下来后，团队协作会更像“流水线”而不是“随机发挥”。

<a id="tab"></a>
## ⑧ Tab 自动补全实操

Tab 的价值在于“持续微提效”。把它当成高频动作加速器，而不是一键写完器。

### 命中率更高的四个前提

1. 命名有语义；
2. 上下文干净；
3. 先写一个正确样本；
4. 保持项目规则一致（lint/类型/命名）。

### 实操节奏

1. 先写函数签名与输入输出类型；
2. 手动完成第一分支；
3. 再用 Tab 续写其余分支；
4. 每完成一个函数跑最小验证。

### Tab 的 4 条安全边界

- 不直接接受你看不懂的业务分支；
- 不在缺少类型约束时盲接大段补全；
- 不在“临时实验代码”堆积状态下持续补全；
- 不把测试与错误处理完全交给自动补全。

### Tab 提示质量提升技巧

1. 在函数顶部写一行“输入 -> 输出”说明；  
2. 关键分支先手动写出第一条；  
3. 对外接口相关代码优先手写，内部样板再交给 Tab；  
4. 每次接受大段补全后立即运行最小测试。

<a id="cmdk"></a>
## ⑨ Cmd+K 内联编辑

Cmd+K 适合“我知道改哪、也知道改成什么”的小到中型修改。

### 指令模板（建议固定）

```text
目标：把这段函数重构为纯函数，输入输出保持不变。
约束：不得修改对外 API 字段。
保留项：保留现有错误码和日志字段。
验收：给出最小测试清单。
```

### 三种常用场景

- 遗留函数可读性重构；
- TypeScript 类型补齐（尽量避免 any）；
- 统一错误处理结构。

改完至少要看一次 diff，再跑最小测试，避免“看起来更优雅、行为却变了”。

### 常用 Prompt 模板（可复用）

```text
将这段函数改为“早返回”结构，降低嵌套层级；
保持输入输出与错误码完全兼容；
不得修改对外接口字段；
改完给出 3 条最小回归测试点。
```

```text
把这段逻辑拆成两个私有函数：参数校验和核心计算；
保留所有日志字段；
禁止新增第三方依赖；
最后输出一段变更说明，便于写 commit message。
```

### Cmd+K 适合“低风险高收益”的改动

- 命名统一（变量名/函数名一致化）；  
- 条件分支可读性优化（早返回、减少嵌套）；  
- 错误处理结构标准化；  
- 注释和类型补全。

不建议把“跨 10+ 文件的大改造”直接交给 Cmd+K，这类任务应交给 Composer/Agent 分批完成。

<a id="chat"></a>
## ⑩ Chat 与 @ 引用

Chat 是“推理中枢”。你给它的上下文越精准，产出越稳定。

### @ 引用怎么用

- `@文件`：定位具体实现；
- `@目录`：评估结构影响；
- `@符号`：聚焦单一函数/类；
- `@报错`：做根因分析。

### 三段式提问法

1. 目标：我要什么结果；
2. 约束：哪些不能改；
3. 输出格式：步骤 / 风险 / 验证命令。

### Chat 里最容易踩的坑

1. **只说“帮我优化”**：目标太泛，结果就泛。  
2. **不给约束**：容易出现越界建议。  
3. **不给输出格式**：难以直接执行。  
4. **一次塞太多问题**：上下文变噪声，回答质量下降。

### Chat 输出结构建议

你可以固定让 Chat 按下面结构输出，便于直接执行：

1. 结论（1 段）；  
2. 执行步骤（3~7 条）；  
3. 风险点（2~4 条）；  
4. 验证命令（可直接复制）。

这比“散文式答案”更适合工程场景，也方便团队复盘。

<a id="composer"></a>
## ⑪ Composer 多文件编辑

当任务跨 3 个以上文件时，Composer 往往比手动切文件稳定。

### 推荐任务描述结构

```text
目标：统一支付模块错误 schema。
影响面：services/payment、api/routes、tests/payment。
禁改项：数据库 schema 不改、外部字段不改。
验收：测试通过，错误码映射保持兼容。
```

### 安全执行策略

1. 先要“计划草案”；
2. 分批执行（每批 3~5 文件）；
3. 每批后看 diff；
4. 偏离就立刻中断重述约束。

### Composer 任务模板（建议收藏）

```text
目标：统一用户模块错误响应结构。
影响面：services/user, api/user, tests/user。
禁改项：数据库 schema、公开 API 字段保持不变。
验收项：单元测试通过；旧错误码映射兼容；新增边界测试 2 条。
输出要求：先给执行计划，再开始改。
```

### Composer 失败时的回滚策略

1. 立刻停止继续批次；  
2. 对最后一批改动做差异审查；  
3. 回滚当前批次并保留前一批稳定结果；  
4. 缩小任务范围后再次执行。

只要你坚持“批次化 + 可回滚”，即使任务复杂也能把风险控制在可接受范围。

<a id="agent"></a>
## ⑫ Agent 模式与 Debug Mode

Agent 适合可分阶段验收的复杂任务。Cursor 官方文档对 Debug Mode 的核心流程定义非常清晰：先探索并提出多个假设，再添加日志埋点收集运行时证据，然后让用户复现问题，基于日志定位根因，最后做有针对性的修复并清理埋点。

你可以把它理解成“证据驱动排障”而不是“凭感觉改代码”。

### 何时优先用 Debug Mode

- bug 可以复现但原因不清楚；
- 竞态/时序问题；
- 性能问题和内存泄漏；
- 回归问题（之前可用、现在失效）。

### 建议工作流

1. 先给完整复现路径；
2. 按 Agent 给的步骤复现；
3. 复现后让 Agent 基于日志定位；
4. 修复后回归验证并清理埋点。

### Debug Mode 的价值边界

- **适合**：根因不清、需要运行时证据的问题；  
- **不适合**：你还没定义好目标、连复现路径都不清楚的问题。

如果连“如何触发 bug”都说不清，建议先在 Chat 模式下把复现步骤标准化，再进入 Debug Mode。

### Debug Mode 复现描述模板

```text
问题：按钮点击后状态显示成功，但数据库没有更新。
复现路径：进入 A 页面 -> 输入测试数据 -> 点击保存。
期望结果：页面成功且数据库新增记录。
实际结果：页面成功，但数据库无记录。
发生频率：10 次中约 7 次。
最近变更：上周新增了异步队列写入。
```

把复现写成这个格式，Agent 更容易建立有效假设并快速定位根因。

<a id="rules"></a>
## ⑬ `.cursorrules` 与 `AGENTS.md` 编写要点

Cursor 官方文档把规则分为项目规则、用户规则、团队规则和 `AGENTS.md`。对大多数项目，先把“可执行规则”写清楚，比反复口头说明有效得多。

| 文件 | 作用 | 适合内容 |
| --- | --- | --- |
| `.cursorrules` 或 `.cursor/rules` | 编码约束 | 命名、类型、目录边界、测试要求 |
| `AGENTS.md` | 协作流程 | 任务拆解、审查步骤、回滚策略 |

### 好规则的标准

- 可执行、可检查；
- 少而精，避免大而全；
- 与 PR 审查项闭环；
- 发现重复错误时再补新规则。

### 两种规则文件怎么分工更稳

- `.cursorrules`：偏代码风格和硬约束；  
- `AGENTS.md`：偏任务流程和协作约定。

不要把它们写成“文学说明文”，要写成“检查清单”，这样模型和人都能执行。

### 可直接复用的最小规则清单

**编码规则**

- 新增 API 必须做输入校验；  
- 禁止改动 `generated/` 目录；  
- 新增 bugfix 必须补至少一条测试。

**协作规则**

- 先计划后执行；  
- 每批改动后要跑测试；  
- 不确定时先提问而不是直接大改。

<a id="mcp"></a>
## ⑭ MCP 配置入门

Cursor 官方将 MCP 定义为连接外部工具和数据源的协议层。实操上，你把它理解成“把外部能力接到当前开发上下文里”就够了。

### 入门四步

1. 明确要接入的工具类型；
2. 用 `mcp.json` 配置服务器（本地或远程）；
3. 先跑只读工具验证；
4. 再逐步开放写操作并做权限最小化。

### 生产建议

- 默认最小权限；
- 敏感工具要求人工确认；
- 关键操作保留审计日志；
- 周期性复查 token 与权限。

### MCP 接入前的三问

1. 这个工具真的需要写权限吗？  
2. 失败后有无回滚路径？  
3. 团队是否有审计留痕机制？

只要有一个问题答不上来，就先保持只读模式。

### MCP 落地的“先易后难”路径

1. 先接入只读检索类工具；  
2. 再接入低风险写操作工具；  
3. 最后再接入高风险或生产写操作工具。

这个顺序能显著降低“第一次接入就踩安全坑”的概率。

## 第三部分 · 故障排查与 FAQ

<a id="fix-login"></a>
## ⑯ 登录与账号问题（5 条）

**Q1：登录后反复跳回登录按钮？**  
A：先退出登录、清理会话缓存、重启客户端；再用网页端确认账号是否正常。

**Q2：验证码邮件收不到？**  
A：先查垃圾邮箱和拦截规则，必要时换常见国际邮箱。

**Q3：提示账号异常或二次验证？**  
A：短期频繁切换网络出口会提升风控概率，先保持同一网络环境完成一次完整登录。

**Q4：已登录但工作区离线？**  
A：先区分“账号可用”与“模型链路可用”，再看模型切换和网络连通性。

**Q5：多设备频繁掉线？**  
A：减少并发登录设备，主力开发集中在一个终端更稳。

### 登录问题排查统一流程（3 分钟版）

1. 确认账号状态（是否可登录、是否被风控）；  
2. 清理会话缓存并重启；  
3. 保持单一网络出口重试；  
4. 记录错误文案与时间点；  
5. 必要时提交支持并附复现步骤。

### 账号问题预防清单

- 减少高频切换网络出口；  
- 不在多台设备同时进行高频操作；  
- 重要操作前先确认账号状态和套餐状态；  
- 保留关键错误信息和时间点，便于后续追踪。

<a id="fix-network"></a>
## ⑰ 网络与节点问题（5 条）

**Q1：客户端能打开，但模型 timeout？**  
A：先切 Composer 2 做对照；如果只在高阶模型超时，优先排查网络链路。

**Q2：频繁 429 是账号坏了吗？**  
A：通常不是。多与并发、请求节奏、任务长度有关，先降并发并拆任务。

**Q3：晚上明显更慢？**  
A：可能是链路拥塞，建议做时段对照并记录失败窗口。

**Q4：Background Agent 排队严重？**  
A：先缩小任务范围并分批提交，避免一次塞入过多文件。

**Q5：网页后台不稳会影响本地吗？**  
A：会影响管理动作，但不一定影响本地编辑，可分开处理。

### 网络问题最小闭环

- 先做模型对照（Composer 2 vs 目标模型）；  
- 再做时段对照（非高峰 vs 高峰）；  
- 再做路径对照（当前网络 vs 备用网络）。

三次对照后你基本能判断问题是在模型、网络还是请求节奏。

### 网络抖动时的应急策略

1. 先降并发和任务体量；  
2. 临时切回 Composer 2/Auto 保证连续输出；  
3. 高阶模型任务延后到网络稳定窗口；  
4. 记录失败窗口，便于后续运营和线路优化。

<a id="fix-models"></a>
## ⑱ 模型与配额问题（5 条）

**Q1：Auto 时好时坏？**  
A：Auto 是策略入口，不是固定单模型，任务复杂度和上下文都会影响输出。

**Q2：配额看着没用完却超限？**  
A：不同能力池可能独立计量，建议结合模型类型看剩余额度。

**Q3：长任务中途截断？**  
A：把任务拆为“计划 -> 执行 -> 验证”三段，逐批确认结果。

**Q4：同提示词结果波动大？**  
A：固定输入模板、固定模型、固定上下文范围再做复测。

**Q5：模型不可用怎么办？**  
A：先查套餐权限、网络可达、客户端版本；再切可用模型保障连续工作。

### 配额问题避免策略

1. 长任务拆分成多个可验收阶段；  
2. 关键阶段再切高阶模型；  
3. 回归阶段切回低成本模型；  
4. 给每次任务设置“最大重试次数”防止空耗。

### 配额管理的团队实践

- 给不同任务类型设置模型优先级；  
- 周复盘一次高成本任务，识别可降本环节；  
- 对“无效重试”建立提醒机制，减少空耗请求。

<a id="fix-billing"></a>
## ⑲ 订阅与支付问题（4 条）

**Q1：扣款成功但套餐未生效？**  
A：先刷新账号并重登，给系统同步时间；仍异常就提交订单号与时间线。

**Q2：续费突然失败？**  
A：检查支付工具状态、账单信息一致性、地区限制和自动续费开关。

**Q3：降档/取消何时生效？**  
A：通常在当前计费周期结束后生效，具体以平台账单页为准。

**Q4：退款最常卡在哪里？**  
A：材料不完整。建议提交“订单号 + 扣款凭证 + 问题时间线 + 已尝试动作”。

### 订阅异常排查模板

```text
问题发生时间：
支付方式：
订单号：
当前账号套餐状态：
是否已重登：
是否已二次确认账单页：
错误提示原文：
已尝试动作：
```

### 支付类工单提交建议

- 工单标题写“问题类型 + 时间 + 影响范围”；  
- 描述中先写结果，再写过程；  
- 附图按“支付侧 -> 账号侧 -> 错误提示”顺序提交。

<a id="fix-privacy"></a>
## ⑳ 隐私、企业、教育优惠（3 条）

**Q1：我的私有代码会被公开吗？**  
A：Cursor 官方企业文档强调可通过隐私模式和组织策略做数据治理；在团队场景里，建议统一隐私设置并定期复核权限边界。

**Q2：企业接入前最该先做什么？**  
A：先搭建身份与访问控制（如 SSO/权限分层）、再上审计与支出控制，最后再扩大团队范围。
